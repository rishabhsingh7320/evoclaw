How it works (behind the scenes)
  1. Submit a query — API creates a Session, Message, and Run. Per-session queue rejects if another run is active.
  2. Planning phase — System prompt + query → GPT-4o-mini → JSON DAG of tasks.
  3. Tool policy — Only tools allowed by profile/allow/deny config are available to the LLM.
  4. Execution phase — Tasks run in topological order (or parallel by depth). For each task, a mini agent loop runs.
  5. Hooks fire — before_tool_call and after_tool_call hooks run around every tool invocation.
  6. Loop detection — If the LLM calls the same tool with the same args 3+ times, a warning is injected.
  7. Context compaction — When prior-task context exceeds the budget, older results are summarized via LLM.
  8. Sub-runs — LLM can delegate a subtask to a child run (no nesting). Child results are injected back.
  9. Memory — Key-value store persists across runs in the same session.
  10. Events stream live — Every action is stored in DB and pushed over WebSocket.
